# grokchat

Простой p2p-чат на Rust + libp2p.

> Объяснение «как для новичка»: этот README написан так, чтобы можно было
> последовательно пройти путь от нуля (что за проект) до диагностики проблем
> (почему не видно пиров, почему не доставляются сообщения).

Этот проект показывает, как несколько программ (нод) находят друг друга в локальной сети и обмениваются сообщениями без центрального сервера.

---

## 1) Что это такое (очень простыми словами)

Этот раздел специально написан без сложных терминов. Если ты видишь libp2p впервые,
прочитай только этот блок и блоки 6/7 — этого уже хватит, чтобы запустить и дебажить MVP.

Есть несколько запущенных копий `grokchat`:

- каждая копия = отдельный **узел (peer)**;
- узел сам открывает порт и слушает входящие подключения;
- узлы ищут друг друга через **mDNS** (внутри локальной сети);
- после подключения они подписываются на общий топик `grok-chat`;
- когда ты пишешь сообщение в консоль, оно рассылается другим подписанным узлам.

То есть это не «клиент-сервер», а **peer-to-peer**: каждый и клиент, и сервер одновременно.

## 2) Стек протоколов и зачем каждый нужен

Ниже порядок «снизу вверх».

Дополнительная подсказка:
- ниже перечислены не «альтернативы», а слои, которые работают вместе;
- если ломается нижний слой (например, TCP-доступность), верхние (gossipsub/mdns)
  тоже будут выглядеть как «не работают».

### TCP

Базовый транспорт. По TCP байты физически едут между двумя нодами.

Практически: если порт закрыт firewall, то до gossipsub дело не дойдёт вообще.

### Noise

Шифрование и аутентификация соединения.

- Шифрует трафик (чтобы его не читали по пути).
- Делает криптографическое рукопожатие между пирами.

Практически: даже в локальной сети это полезно, чтобы сообщения не ехали открытым текстом.

### Yamux

Мультиплексирование потоков поверх одного TCP-соединения.

- Представь один провод (TCP), а внутри несколько «виртуальных каналов».
- Это экономит соединения и упрощает одновременную работу нескольких протоколов.

Практически: без мультиплексирования пришлось бы плодить больше отдельных соединений.

### mDNS

Автопоиск пиров в локальной сети (LAN).

- Узлы объявляют себя и слышат соседей по broadcast/multicast.
- Работает только в пределах локальной сети (обычно один сегмент LAN).

Важно: mDNS не равно «интернет-discovery»; для интернета нужен bootstrap/relay/NAT traversal.

### Gossipsub (PubSub)

Протокол рассылки сообщений по топикам.

- Топик в проекте: `grok-chat`.
- Нода подписывается на топик и получает сообщения от других подписчиков.
- Включён `flood_publish(true)`, поэтому публикация может пройти даже при временно пустом mesh.

Важно: `flood_publish(true)` повышает шанс доставки в маленькой и нестабильной сети,
но это не заменяет нормальное формирование mesh.

## 3) Как устроен обмен сообщениями в этом проекте

Ниже «линейка старта»: если что-то не работает, смотри, на каком шаге останавливается процесс.

1. Программа стартует.
2. Загружает конфиг `grokchat.conf` (если есть).
3. Создаёт новый `PeerId` (из новой ed25519-ключ-пары в рантайме).
4. Поднимает transport: `TCP -> Noise -> Yamux`.
5. Включает два поведения libp2p:
   - `mdns` для discovery;
   - `gossipsub` для чата.
6. Подписывается на топик `grok-chat`.
7. Начинает слушать порт (`--port` или случайный при `0`).
8. Если задан peer (`--peer` или из конфига) — делает dial.
9. По mDNS находит соседей и пытается подключиться (с защитой от dial-штормов).
10. При вводе строки в stdin делает `publish` в `grok-chat`.

## 4) Важные параметры Gossipsub (что ты настроил и почему)

Этот раздел — про причины выбранных значений, а не «магические числа».
Для продакшн-сценариев параметры обычно делают более консервативными.

В `main.rs` используются:

- `heartbeat_interval(100ms)` — частые heartbeat для быстрого «прогрева» mesh;
- `mesh_n_low(1)`, `mesh_n(2)`, `mesh_outbound_min(1)` — мягкие настройки под маленькое число узлов;
- `validation_mode(Permissive)` — не строгая валидация;
- `flood_publish(true)` — публикация может отправляться через известных пиров, даже если mesh временно пуст.

Также есть кастомный `message_id_fn`: ID сообщения считается по хэшу `message.data`, чтобы дубликаты одинакового payload лучше отсеивались.

## 5) Конфиг и аргументы запуска

Зачем это нужно:
- чтобы не вводить bootstrap/port каждый запуск вручную;
- чтобы MVP вёл себя предсказуемо между перезапусками.

### Файл `grokchat.conf`

Пример:

```ini
# grokchat startup config
bootstrap_addr=/ip4/192.168.1.10/tcp/4001/p2p/12D3KooW...
listen_port=4001
```

### Аргументы CLI

- `--peer <MULTIADDR>` — bootstrap-адрес пира;
- `--port <PORT>` — локальный порт прослушивания;
- также поддерживается позиционный аргумент peer (без `--peer`).

При первом запуске, а также при передаче `--peer`/`--port`, значения сохраняются в `grokchat.conf`.

## 6) Как запустить (пошагово, «с нуля»)

Мини-совет: для первых тестов лучше открыть два терминала на одной машине
и запустить две ноды на разных портах. Так проще понять базовый поток событий.

## Требования

- Rust toolchain (`rustup`, `cargo`);
- 2+ терминала (или 2+ машины в одной LAN).

## Сборка

```bash
cargo build
```

## Запуск первого узла

```bash
cargo run -- --port 4001
```

Смотри в вывод:

- `🔐 Твой Peer ID: ...`
- `🎧 Слушаю на: /ip4/.../tcp/4001`

Возьми полный multiaddr вида:

`/ip4/192.168.1.10/tcp/4001/p2p/<PeerId>`

## Запуск второго узла с подключением к первому

```bash
cargo run -- --port 4002 --peer /ip4/192.168.1.10/tcp/4001/p2p/<PeerId>
```

После установления связи увидишь события контакта/подписки. Теперь ввод текста в любом узле должен приходить на другой.

## 7) Что означают частые сообщения в логах

Этот блок нужен как «словарик диагностики»: смотри совпадение симптома и значения лога.

### `mesh=0`, но сообщение отправилось

Это нормально при `flood_publish(true)`.

- `mesh` — динамическое подмножество соседей для gossip;
- `known` peers могут быть > 0;
- публикация может пройти через flood-маршрут.

### `InsufficientPeers`

В момент отправки не хватило подходящих связей. Код уже пробует «подтолкнуть» отправку и повторить.

Если это происходит постоянно — значит сеть не стабилизируется (проверь discovery и firewall).

### mDNS нашёл, но связи нестабильны

Обычно это:

- разные сети/VPN/изоляция между подсетями;
- firewall блокирует порт;
- временный churn соединений.

## 8) Логи и диагностика

Рекомендация: при проблемах всегда сначала открывай лог и ищи последовательность:
discovery -> dial -> connection established -> subscribed -> publish/receive.

- Файл лога: `grokchat.log`;
- Логируются:
  - discovery (`mDNS discovered/expired`),
  - establish/close соединений,
  - подписки/отписки,
  - попытки publish,
  - состояние mesh и known peers.

Это позволяет понять, где именно проблема: discovery, dial, подписка или доставка.

## 9) Ограничения текущей реализации

Этот раздел специально оставлен честным: здесь перечислены ограничения MVP,
которые важно понимать до расширения проекта.

- `PeerId` генерируется заново на каждом старте (идентичность не постоянная).
- mDNS работает в локальной сети, не для интернета.
- Нет NAT traversal/relay механик для сложных сетей.
- Нет истории сообщений/персистентного хранилища.

## 10) Быстрый FAQ

Если вопроса нет в списке, проверь сначала блок 7 (логи), затем блок 8 (диагностика).

### Почему не вижу пиров?

Проверь, что узлы в одной LAN, порты открыты, firewall не режет входящие.

### Почему адрес меняется?

Если порт `0`, ОС выдаёт случайный порт. Задай `--port` явно.

### Можно ли подключаться без mDNS?

Да. Передай `--peer` (или позиционный peer-адрес). Это bootstrap вручную.
