# Архитектура mobile libp2p

Этот документ описывает целевую схему мобильной p2p-ноды простым инженерным языком:
- что за компоненты есть;
- как данные идут между слоями;
- где зона ответственности каждого слоя.

## 1. Компоненты

Идея разделения: UI не должен знать детали сетевого стека, а core не должен знать UI-фреймворк.

- `mobile-core` (Rust)
  - libp2p transport
  - gossipsub
  - kademlia
  - (позже) request/response для chunks
- `mobile-bridge`
  - адаптер между Rust и нативным UI runtime
- `mobile-ui`
  - чат, профиль, статус сети

Пояснение:
- `mobile-core` — «двигатель сети»;
- `mobile-bridge` — «переходник платформы»;
- `mobile-ui` — «экран и взаимодействие с пользователем».

## 2. Потоки данных

Раздел специально показывает два направления отдельно, чтобы не путать командный и событийный потоки.

### От UI к сети

`UI -> BridgeCommand::SendChat -> core -> gossipsub.publish`

То есть UI не публикует в сеть напрямую: он отправляет типизированную команду в core.

### От сети к UI

`libp2p event -> core -> CoreEvent::ChatMessage/Presence/Profile -> UI`

То есть любые сетевые события сначала нормализуются в CoreEvent и только потом идут в интерфейс.

## 3. Идентичность

Цель этого блока: закрепить стабильную личность пользователя в p2p-сети между перезапусками.

- На первом запуске генерируется keypair
- Keypair сохраняется в secure storage платформы
- При следующем запуске восстанавливается тот же PeerId

Важно: если keypair теряется, PeerId поменяется и узел станет «новым» для сети.

## 4. Синхронизация между нодами

Этот блок описывает «живое состояние» в сети: сообщения, онлайн-статус, профиль.

- Канал сообщений: gossipsub topic (`grok-chat`)
- Presence: периодический heartbeat envelope
- Profile: отправка profile envelope на старт/изменение

Практический смысл:
- чат — для контента;
- presence — для наблюдаемости;
- profile — для человекочитаемой идентификации.

## 5. DHT (Kademlia)

Здесь описан слой распределённого поиска/объявления данных и пиров.

- bootstrap peers в настройках
- `GET_PROVIDERS/START_PROVIDING` для blob keys
- хранение известных peers в локальном state

Важно: качество bootstrap-узлов напрямую влияет на скорость «входа в сеть».

## 6. Файлы

Файловый обмен вынесен отдельно, потому что требования к нему отличаются от текстового чата.

- chunking (1 MiB)
- hash (`blake3`)
- manifest c metadata + chunk hashes
- transport chunks через request/response

Логика: сначала манифест, потом адресный запрос недостающих чанков.

## 7. Безопасность

Раздел фиксирует базовый security baseline и возможное усиление в будущем.

- Noise transport
- подпись envelope источником
- (опционально) E2EE payload для приватных чатов

Минимальная цель MVP: защищённый транспорт и проверяемый источник сообщений.
