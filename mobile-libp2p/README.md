# mobile-libp2p (план + скелет)

Цель: каждый мобильный пользователь = отдельный libp2p-узел (свой `PeerId` / DHT-участник), без зависимости от gateway как «заменителя пира».

Этот README описывает стратегическое направление:
- зачем нужен отдельный mobile core;
- что уже подготовлено;
- в каком порядке лучше внедрять реальные сетевые функции.

## Что уже есть в этом скелете

Это важно понимать как «точку старта», а не как «полностью готовый продукт».

- Документ архитектуры и этапов внедрения
- Rust core crate (`mobile-libp2p/core`) с базовыми типами команд/событий
- Контракт для bridge-слоя (Android/iOS)

Практически: документация уже задаёт единый язык между backend/mobile/platform командами.

## Этапы реализации

Этапы перечислены в рекомендуемом порядке, чтобы минимизировать риск пересборки архитектуры.

1. Core runtime (Rust): старт узла, keypair, события сети
2. Gossipsub: join topic, send/recv message
3. Kademlia: bootstrap + provider lookup
4. Blob-store: chunk API (put/get)
5. Bridge: UniFFI/JNI/Swift bindings
6. Mobile app: React Native (native module) или Flutter

Почему так:
- сначала устойчивый core runtime;
- потом realtime обмен;
- потом discovery/данные;
- затем платформенный bridge и финальная UI-интеграция.

## Почему не Expo Go

Раздел про ограничение среды выполнения.

`Expo Go` не запускает ваш произвольный Rust native core с libp2p.
Нужен prebuild/dev client (или bare RN), где подключается нативный модуль.

Иными словами: для настоящего mobile libp2p нужна сборка с нативными бинарями,
а не только JavaScript-слой.

## Что делать дальше

Ниже — практический чеклист следующего шага внедрения.

- Реализовать `start_node()` и сетевой loop в `core/src/lib.rs`
- Добавить `tokio + libp2p` зависимости в `core/Cargo.toml`
- Выбрать bridge:
  - RN: TurboModule + FFI
  - Flutter: rust_bridge

Рекомендация:
- зафиксировать целевую платформу bridge заранее (RN или Flutter),
  чтобы не вести две интеграции одновременно на ранней стадии.
